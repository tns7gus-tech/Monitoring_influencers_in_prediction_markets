# Polymarket 인플루언서 모니터링 웹 기획서

작성일: 2026-03-14  
대상: Polymarket 전용 MVP  
문서 목적: 상위 트레이더 추적, 성과 분석, 카피베팅 시뮬레이션을 위한 서비스 기획 정리

---

## 1. 프로젝트 개요

본 서비스는 Polymarket의 공개 시장 데이터와 지갑/주소 기반 활동 데이터를 활용하여, 예측시장에서 성과가 높은 트레이더를 추적하고 모니터링하는 웹 서비스다.

초기 목표는 실제 자동매매가 아니라 다음 기능을 제공하는 것이다.

- 성과가 좋은 트레이더 탐색
- 주소별 거래 성향 분석
- 승률, 수익률, 최근 성과 랭킹 제공
- 특정 날짜부터 해당 트레이더를 따라갔을 경우의 가상 수익률 분석
- 진입/청산 타이밍 패턴 시각화

중장기적으로는 사용자 본인 계정 연동 후 반자동 또는 자동 카피베팅 기능까지 확장할 수 있다.

---

## 2. 핵심 결론: API가 사용자별 발급인가?

Polymarket 관련 API는 성격상 크게 2종류로 나뉜다.

### 2.1 공개 조회용 API
이 영역은 일반적으로 사용자별 발급 없이 활용 가능한 범주로 본다.

주요 활용 예시:
- 시장 목록 조회
- 이벤트 정보 조회
- 가격 / 오더북 조회
- 체결 내역 / 시장 상태 조회
- 특정 주소 활동 모니터링
- 랭킹 계산용 데이터 수집

즉, 본 프로젝트의 **모니터링 웹 MVP**는 대체로 사용자별 API 발급 없이 시작하는 방향이 적합하다.

### 2.2 사용자 인증이 필요한 거래용 API
이 영역은 각 사용자 본인의 인증 정보가 필요하다.

주요 활용 예시:
- 주문 생성
- 주문 취소
- 실제 카피베팅 실행
- 계정 잔고 / 포지션 동기화
- 사용자별 거래 자동화

즉, **실제 베팅 기능**을 넣는 순간 각 사용자는 본인 지갑 또는 본인 인증 정보를 통해 연결해야 한다.

## 3. 서비스 목표

### 3.1 1차 목표 (MVP)
- Polymarket 상위 트레이더 모니터링
- 주소별 성과 추정
- 트레이더별 성향 분석
- 가상 카피베팅 백테스트
- 리더보드 및 상세 페이지 제공

### 3.2 2차 목표
- 상위 트레이더의 신규 진입 알림
- 특정 조건 만족 시 텔레그램/디스코드 알림
- 내 관심 트레이더 즐겨찾기 및 비교

### 3.3 3차 목표
- 사용자 지갑 연동
- 조건부 실제 카피베팅
- 손절/익절/최대베팅 한도 관리
- 포트폴리오 리스크 제어

## 4. 사용자 시나리오

### 4.1 방문자
- 오늘 성과 좋은 트레이더를 본다.
- 최근 30일 수익률 상위 계정을 본다.
- 특정 트레이더 상세 페이지에서 어떤 시장을 주로 맞추는지 본다.
- “30일 전부터 따라갔으면 얼마 벌었는지”를 확인한다.

### 4.2 고급 사용자
- 여러 트레이더를 비교한다.
- 트레이더별 진입 타이밍 차이를 본다.
- 특정 카테고리(정치, 거시경제, 스포츠 등)별 적중률을 확인한다.
- 특정 주소가 새 포지션에 진입하면 알림 받는다.

### 4.3 향후 유료 사용자
- 자신의 지갑을 연결한다.
- 특정 트레이더를 일정 비율로 카피하도록 설정한다.
- 최대 노출 금액, 슬리피지 한도, 추종 중단 조건을 설정한다.

### F. 시각화
- 트레이더 상세 페이지
- 누적 수익률 차트
- 승률 및 거래 횟수 차트

## 8. 승률 높은 사용자 판별 로직

단순 승률만으로 고수를 판별하면 왜곡이 생긴다. 따라서 아래 요소를 함께 계산해야 한다.

### 8.1 기본 지표
- 총 거래 수
- 승 거래 수 / 패 거래 수
- 승률
- 평균 수익률
- 누적 손익

### 8.2 보정 지표
- 최소 거래 수 필터
- 최근 활동 가중치
- 시장 유동성 가중치
- 포지션 규모 반영
- 카테고리 집중도 반영
- 변동성 조정 성과

### 8.3 추천 랭킹 공식 예시
랭킹 점수 =  
(최근 90일 수익률 × 0.35) +  
(최근 30일 수익률 × 0.25) +  
(승률 × 0.15) +  
(거래 수 신뢰도 점수 × 0.10) +  
(변동성 조정 점수 × 0.15)

### 8.4 제외 조건 예시
- 거래 수 지나치게 적은 주소
- 종료 안 된 포지션만 많고 실현 성과가 불분명한 주소
- 특정 저유동성 시장에서만 성과가 나온 주소
- 단발성 대박으로 왜곡된 주소

## 9. 카피베팅 백테스트 설계

## 9.1 목적
“이 트레이더를 특정 날짜부터 일정 규칙으로 따라갔으면 얼마나 벌었는가?”를 계산한다.

## 9.2 입력값
- 기준 트레이더 주소
- 추종 시작일
- 투자금
- 추종 비율
- 최소/최대 포지션 크기
- 동일 시장 중복진입 허용 여부
- 수수료/슬리피지 가정

## 9.3 기본 시뮬레이션 규칙 예시
- 트레이더가 새 포지션 진입 시 동일 방향으로 추종
- 일정 지연시간 반영
- 비율에 맞춰 모의 진입
- 종료 또는 청산 신호 시 모의 청산
- 누적 수익률 계산

## 9.4 비교 지표
- 단순 추종 수익률
- 동일 기간 시장 평균 대비 초과성과
- 최대 낙폭(MDD)
- 승률
- 샤프 유사 지표
- 월별 수익률 분포

## 9.5 주의사항
- 실제 체결 가능 가격과 차이가 날 수 있음
- 유동성 부족 시장에서는 이론 수익률이 과대추정될 수 있음
- 후행 추종 시 성과가 급격히 악화될 수 있음

## 15. 주요 리스크

### 15.1 데이터 해석 리스크
- 주소 단위 성과가 실제 사용자 실력과 완전히 일치하지 않을 수 있음
- 다지갑 사용 가능성
- 미실현 포지션 해석 문제

### 15.2 실행 리스크
- 후행 추종 시 성과 저하
- 낮은 유동성에서 복제 불가
- 실제 체결가와 백테스트 괴리

### 15.3 정책/규제 리스크
- 실거래 자동화는 지역/규제/서비스 정책 검토 필요
- 카피트레이딩 성격의 기능은 법적 검토가 필요할 수 있음

---

## 16. 현실적인 권장 방향

가장 좋은 시작 방식은 아래와 같다.

### 추천 전략
1. **모니터링 웹만 먼저 개발**  
2. **공개 데이터 기반으로 고수 탐색과 백테스트부터 구현**  
3. **실제 거래 연동은 나중에 분리**

이렇게 해야 개발 난이도와 운영 리스크를 동시에 줄일 수 있다.

## 17. 최종 정리

- Polymarket 전체 API가 사용자별 발급인 것은 아니다.
- 조회/분석용 데이터는 사용자별 발급 없이 시작 가능한 영역이 있다.
- 하지만 실제 주문/카피베팅은 각 사용자별 인증이 필요하다.
- 따라서 본 프로젝트는 **무인증 모니터링 MVP → 사용자 인증 기반 실거래 기능 확장** 순서가 가장 합리적이다.

## 18. 확인 메모

이 문서는 현재 대화 기준으로 서비스 기획 관점에서 정리한 초안이다.  
이번 세션에서는 공식 문서 웹 확인 도구가 정상 동작하지 않아 최신 엔드포인트 이름이나 인증 절차를 직접 검증하지 못했다.  
따라서 실제 개발 착수 전에는 Polymarket 공식 개발자 문서 기준으로 다음 항목을 한 번 더 검증해야 한다.

- 최신 엔드포인트 경로
- 공개 조회 가능 범위
- 인증 필요 엔드포인트 범위
- 속도 제한(rate limit)
- API 이용 정책 및 변경 사항

## 19. 현재 MVP 구현 반영 메모 (2026-03-14)

이 문서는 Polymarket 전용 MVP 기획 문서이며, 현재 저장소 구현과 연결되는 사항을 아래처럼 정리한다.

### 현재 실제 구현 스택
- Node.js 서버
- 로컬 SQLite 저장소
- 정적 프론트엔드 대시보드
- 수동 동기화 + 주기 동기화

### 현재 실제 구현 기능
- 정확도 중심 트레이더 랭킹
- 최근 포지션 변화 피드
- watchlist 지갑 저장
- 규칙형 알림 생성
- 시장별 최근 가격 차트
- 실거래 기반 카피백테스트

### 현재 구현되지 않은 범위
- 사용자 인증
- 실제 주문 생성/취소
- 자동 카피베팅 실행
- 외부 알림 채널 연동

### 현재 API 기준 보완 사항
내부 서비스 API는 기획서의 추상 설계보다 단순하며, 아래 엔드포인트가 실제 기준이다.
- `GET /api/bootstrap`
- `POST /api/sync`
- `GET /api/watchlist`
- `POST /api/watchlist`
- `DELETE /api/watchlist/:wallet`
- `GET /api/alerts`
- `POST /api/alerts/read-all`
- `GET /api/markets`
- `GET /api/markets/:slug`
- `GET /api/traders/:id/backtest`

### 후속 개발 우선순위
- SQLite 스냅샷 이력을 기간별 테이블로 확장
- 시장별 시계열 캐시 고도화
- 사용자 계정/권한 모델 추가
- Kalshi를 별도 모듈로 분리 설계
